# Drupal Commerce 3: From Modules to Working Store

You've got the modules installed and enabled. Now comes the part that's actually Commerce-specific: the entity architecture. This is the bit that trips people up coming from Shopify/WooCommerce, so I'll explain the *why* at each stage, not just the *click here*.

---

## The mental model first

Commerce splits "a product" into two entities, and this is the single most important concept to internalize:

- **Product Variation** — the thing that's actually purchasable. Has a SKU, a price, and stock. A t-shirt in "Blue, Large" is a variation.
- **Product** — the marketing wrapper. Has a title, body, images. A t-shirt is a product; it groups all its variations together (Blue/Large, Blue/Medium, Red/Large...) so they display as one page with dropdowns.

Everything else (Attributes, Product Types, Variation Types) exists to define the *shape* of these two entities before you start creating real data.

Order of setup: **Store → Attributes → Variation Type → Product Type → actual Products → Payment Gateway → Checkout Flow → Shipping/Stock**. Do it in this order or you'll be backfilling config onto live data.

---

## Step 1: Create your Store

A Store in Commerce is a config entity — it defines currency, country, and which product types it sells. Even a single-store site needs exactly one of these to exist before products can be created.

Go to: `/admin/commerce/config/stores` → **Add a new store**

Fill in:
- **Name**: e.g. "Main Store"
- **Store owner**: whichever user should show as the order placer for admin-created orders
- **Default currency**: GBP (or whatever applies)
- **Address**: your registered business address — this is used for tax/shipping origin calculations later

Then set it as default at `/admin/commerce/config/stores` (the "Default store" gets auto-assigned to new orders/products when there's only one store, but confirm it).

Check it landed correctly:
```
ddev drush php:eval "print_r(\Drupal::entityTypeManager()->getStorage('commerce_store')->loadMultiple());"
```

---

## Step 2: Define Attributes (if you sell variants)

Skip this step entirely if every product you sell has no variants (e.g. a single-SKU digital download, a membership). Only do this if you need things like Size/Color pickers.

Go to: `/admin/commerce/product-attributes` → **Add attribute**

Create one attribute per axis of variation — e.g. "Size" and "Color" as two *separate* attributes, not one combined field. Each attribute then gets its own set of values (e.g. Size → Small, Medium, Large).

This is the step your setup doc flagged as a common mistake: Size should be a **Commerce Attribute**, not a plain entity reference or list field bolted onto the variation. Attributes are what power the automatic "Select options" widget on the product display and keep variation combinations unique.

---

## Step 3: Configure a Product Variation Type

Go to: `/admin/commerce/config/product-variation-types`

The default `Default` variation type exists out of the box and is fine to start with, or clone it if you want a separate type per product category (e.g. "Clothing" vs "Digital").

Edit your variation type and:
1. Under **Edit** → confirm **SKU**, **Price**, and **Status** fields are present (they are by default)
2. Under **Manage fields** → add any variation-specific fields (weight/dimensions if you enabled `commerce_shipping`, since shipping calculations read from the variation, not the product)
3. Under **Manage fields** → if you created Attributes in Step 2, attach them here as attribute fields (Commerce usually prompts you to do this from the attribute screen directly — "Used in variation type" checkbox)

---

## Step 4: Configure a Product Type

Go to: `/admin/commerce/config/product-types`

Again, `Default` exists already. Edit it (or add a new one) and confirm:
- **Variation type**: linked to the one you set up in Step 3
- **Multiple variations**: enable this if products can have variants (Size/Color combos); leave off for single-SKU products

Under **Manage fields** for the Product Type, add your marketing-facing fields: body/description, image gallery, related products, etc. This is separate from the variation's field list — keep pricing/SKU/stock fields on the *variation*, and descriptive/marketing fields on the *product*.

---

## Step 5: Create your first real product

Go to: `/admin/commerce/products` → **Add product**

- Fill in title, body, images (product-level fields from Step 4)
- Add at least one variation: SKU, price, and (if applicable) attribute values like Size/Color
- If "Multiple variations" was enabled, use **Add variation** to add Blue/Large, Blue/Medium, etc. — each needs its own SKU and price
- Set **Published** and **Store** (should auto-populate if you only have one store)

Verify it's live:
```
ddev drush php:eval "print_r(\Drupal::entityTypeManager()->getStorage('commerce_product')->loadMultiple());"
```

Then visit the product's canonical URL and confirm the Add to Cart form renders with your attribute dropdowns (if any).

---

## Step 6: Wire up a payment gateway

For learning purposes, skip Stripe/PayPal entirely at first — use the built-in **Manual** gateway. No API keys, no external account, no HTTPS requirement. It just marks the order as "pending" so you can walk the full checkout flow end to end without any payment-processor complexity in the way.

Go to: `/admin/commerce/config/payment-gateways` → **Add payment gateway**

- **Plugin**: `Manual`
- **Name**: e.g. "Test — Manual"
- Leave everything else default, save

On checkout, this shows as a selectable payment method with no card fields at all — the customer places the order and it lands as "pending," which you confirm manually later from `/admin/commerce/orders`.

**Only move to Stripe once you're ready to test real payment UI.** If/when you do:

- **Plugin**: Stripe (Payment Intents API, not Stripe Checkout)
- Enter **test mode** publishable/secret keys — never go live until checkout is fully tested end to end
- **Never commit the secret key to `config/sync`.** Keep it in `settings.local.php` via `getenv('STRIPE_SECRET_KEY')`, the same pattern you already use for AutoAddress/Dynamics, so it never ends up in an exported `.yml` file and gets caught by GitHub's push protection.
- If the card fields don't render on checkout: check the browser console first. It's almost always either a CSP block on `js.stripe.com`/`api.stripe.com` (add via `ddev drush config:edit csp.settings`) or JS aggregation mangling the script (`ddev drush config:set system.performance js.preprocess 0 -y` for local dev).
- Push through a full test purchase using [Stripe's test card numbers](https://docs.stripe.com/testing) before touching real keys.

---

## Step 7: Review the Checkout Flow

Go to: `/admin/commerce/config/checkout-flows`

The default flow (`Default`) covers Login → Order Information → Review → Payment → Complete. You generally don't need to touch this yet — it's driven by the panes enabled inside it. Worth opening it once now just to see the pane list, since you'll likely be customizing the **Completion** pane later (this is the "thank you" page work most people end up doing).

---

## Step 8: Shipping and Stock (only if physical goods)

**Shipping** — `/admin/commerce/shipping-methods` → Add a shipping method, pick a rate type (flat rate is simplest to start), and a condition (e.g. applies to your one Store). Note this path is *not* nested under `config/` like most other Commerce settings — it's its own top-level collection.

**Stock** — `commerce_stock` needs a Stock Location and a Stock Service configured at `/admin/commerce/config/stock`. Without this configured, the module is enabled but inert — it won't actually track or decrement inventory.

---

## Sanity-check the whole chain

**A heads-up before you do this**: Commerce doesn't ship with a public storefront/catalog page out of the box — there's no default "shop" listing to browse. To view a product as a customer would, go to `/admin/commerce/products`, click into your product, and use the **View** tab/link on that edit screen — it takes you to the real canonical page (e.g. `/product/1`) with the live Add to Cart form. Building an actual `/shop` catalog page is a separate step using Views (Content: Product, filtered to Published) — worth knowing it doesn't exist yet so you're not hunting for something that isn't there.

Once Steps 1–8 are done, run through one full test order manually: open the product's canonical page → add to cart → checkout → Manual payment (or Stripe test-mode) → land on completion page → check the order appears at `/admin/commerce/orders`. That end-to-end pass is the real confirmation your stack works, more than any individual config screen.

---

## What's genuinely next after this

- **Order workflows** (`/admin/commerce/config/order-types`) — control what happens to an order's state (draft → placed → fulfilled)
- **Promotions** (`/admin/commerce/promotions`) — discount codes, if needed
- **Taxes** (`commerce_tax` module, not yet installed) — needed before going live in most jurisdictions
- **Search API index** — since you installed `search_api`/`facets`, you'll want to index the product content type and build a Views-based catalog page with facet filters
